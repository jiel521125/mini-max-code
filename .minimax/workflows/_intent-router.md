# 意图判别规则（双阶段）

> **位置**：`./.minimax/workflows\_intent-router.md`
> **职责**：定义主 agent 收到用户输入后，如何做双阶段意图判别
> **执行时机**：每次用户输入到达时，**阶段 1 和阶段 2 必须同时并行跑**

## 阶段 1：关键词/正则粗筛

**目标**：快速匹配已固化的模板，避免 LLM 推理开销。

**执行步骤**：
1. 扫描 `./.minimax/workflows\INDEX.md` 中所有 active 模板的 `triggers` 字段
2. 对用户输入做**不区分大小写**的关键词匹配
3. **正则匹配**：对 triggers 中含正则语法（`*` `?` `[abc]`）的项启用正则引擎
4. 匹配策略：
   - **任一关键词命中** → 模板命中
   - **多模板同时命中** → 取 `triggers` 关键词匹配数最多的
   - **多模板命中数相同** → 取最近创建的（按 `created_at`）
5. 输出：
   - 命中：`matched_template_id: <id>, matched_keywords: [keywords]`
   - 不命中：`stage1_hit: false`

**性能约束**：必须在 50ms 内完成（纯字符串匹配）。

## 阶段 2：LLM 推理编排

**目标**：当阶段 1 不命中时（或用户明确要求重新编排时），基于意图 + 子 Agent 能力推理出 ≤3 个候选工作流。

**执行步骤**：
1. 主 agent 加载已注册的子 Agent 列表（`./.minimax/agents\registry.json`）
2. 主 agent 加载已固化的模板列表（`./.minimax/workflows\INDEX.md`）
3. 主 agent **推理**：
   - 用户的真实意图是什么？
   - 哪些子 Agent 的 `capabilities` 匹配此意图？
   - 哪些已固化模板可作为"积木"复用？
   - 编排出一条端到端的工作流（步骤、依赖、并行、产出物）
4. 推理产出 **≤3 个候选工作流**，每个候选必须包含：
   - `workflow_draft`: 工作流草稿（按 `_template.md` 的 schema）
   - `reasoning`: 为什么这样编排（用人类语言说明）
   - `sub_agents_used`: 调用的子 Agent 列表
   - `estimated_steps`: 步骤数估计
   - `risks`: 潜在风险

**性能约束**：允许 LLM 推理时间，不设上限（但 5 分钟未出结果则超时重试）。

## 双阶段并行执行

**关键规则**：阶段 1 和阶段 2 **必须同时启动**，不是串行。

```
[用户输入]
    ↓
    ┌─────────────────┐  ┌─────────────────┐
    ↓                 ↓  ↓                 ↓
[阶段 1 启动]         [阶段 2 启动]
(关键词粗筛)          (LLM 推理)
    │                 │
    ↓                 ↓
   <50ms             5min 超时
    │                 │
    ↓                 ↓
[阶段 1 结果]         [阶段 2 结果]
    │                 │
    └────────┬────────┘
             ↓
[主 agent 整合]
   - 阶段 1 命中的固化模板（若有）
   - 阶段 2 推理的 ≤3 个动态候选
   - 标注："[固化模板]" vs "[动态候选]"
             ↓
[展示给用户]
             ↓
[用户选择]
   ├── 选固化模板 → 直接执行
   ├── 选动态候选 → 执行 → 询问是否固化为新模板
   ├── 自定义修改 → 调整 → 可选固化
   └── 全部拒绝 → 询问用户期望什么样的工作流
```

## 整合展示格式

主 agent 整合后的展示**必须**遵循以下格式：

```markdown
## 意图判别结果

### 阶段 1：关键词粗筛
- **结果**: [命中 / 未命中]
- **命中模板**: [template_id]（若有）
- **匹配关键词**: [keywords]（若有）

### 阶段 2：LLM 推理
- **结果**: [产出 N 个候选，N ≤ 3]

#### 候选 1：[名称]
- **编排思路**: [为什么这样编排]
- **子 Agent**: [调用的子 Agent 列表]
- **步骤数**: [N 步]
- **风险点**: [潜在风险]

#### 候选 2：[名称]（若有）
- ...

#### 候选 3：[名称]（若有）
- ...

### 请选择

请告诉我：
1. 用阶段 1 命中的固化模板（若有）？
2. 用阶段 2 的哪个候选？
3. 都不满意？说出你的调整方向
```

## 边界情况

### 阶段 1 和阶段 2 同时无结果

- 主 agent 直接询问用户：需要执行什么类型的任务？
- 提示用户：可补充关键词，或参考 `agents/README.md` 中的子 Agent 列表

### 阶段 1 命中但阶段 2 推荐不同

- **优先展示阶段 1 命中**，但同时展示阶段 2 的对比
- 用户决定

### 阶段 2 候选数 < 3

- 输出推理出的候选即可，不强制凑数
- 若只有 1 个候选，标注"仅有 1 个候选"

### 用户明确"重新编排"

- 跳过阶段 1，直接跑阶段 2
- 在展示结果中标注"用户触发重新编排"

## 强制约束

- **阶段 1 必须在 50ms 内返回结果**（含命中或不命中）
- **阶段 2 最多输出 3 个候选**（超过截断，并标注"截断"）
- **每个候选必须附"为什么这样编排"**——没有理由的候选不接受
- **必须等用户选择后才能执行**——不得擅自选默认候选
- **用户拒绝所有候选时**——必须继续对话澄清，不得强行执行

## 与其他文件的关系

- 依赖 `agents/registry.json`（子 Agent 列表）
- 依赖 `workflows/INDEX.md`（已固化模板）
- 写入 `workflows/<workflow-id>.md`（固化新工作流时）
- 写入 `rules/workflow-decisions.md`（记录固化决策）
