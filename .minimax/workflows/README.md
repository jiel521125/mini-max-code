# 固化工作流模板库

> **位置**：`./.minimax/workflows\`
> **归属**：本项目（天枢项目）专用
> **职责**：存放已锁定的固化工作流模板 + 意图路由规则

## 双模式引擎总览

本项目采用**双模式工作流引擎**：

```
[用户输入]
    ↓
    ┌──────────────────┐  ┌──────────────────┐
    ↓                  ↓  ↓                  ↓
[阶段 1]              [阶段 2]
关键词/正则粗筛         LLM 推理编排
(读各模板的 triggers)   (基于意图+子 Agent 能力)
    │                  │
    ↓                  ↓
  命中固化模板?         产出 ≤3 个候选工作流
    │                  每个候选附"为什么这样编排"
    ↓                  ↓
  命中 → 调固化         不命中 → 跳过
  不命中 → 跳过         输出候选
    │                  │
    └────────┬─────────┘
             ↓
   [主 agent 整合展示给用户]
   - 阶段 1 命中的固化模板（若有）
   - 阶段 2 推理的 ≤3 个动态候选
   - 每个候选的"编排理由"
             ↓
   [用户选择]
   ├── 选固化 → 直接执行
   ├── 选动态候选 → 执行后固化为新模板
   └── 自定义 → 调整后固化
```

**关键点**：
- 阶段 1 和阶段 2 **同时并行跑**，不是串行
- 阶段 2 **必须**输出 ≤3 个候选 + 每个的编排理由
- **只有用户选择后**，动态候选才会固化为模板

## 目录结构

```
workflows/
├── README.md             ← 本文件
├── _template.md          ← 固化工作流的标准 schema
├── _intent-router.md     ← 意图判别规则（双阶段）
├── INDEX.md              ← 已固化模板的索引（自动维护）
└── <workflow-id>.md      ← 具体的固化工作流模板
```

## 文件命名规范

固化工作流文件名格式：`<category>-<action>-<seq>.md`

- `<category>`：分类（feature/bugfix/research/review/deploy/doc）
- `<action>`：动作（develop/fix/investigate/audit/release/write）
- `<seq>`：4 位序号

**示例**：
- `feature-develop-0001.md`（新功能开发工作流 #1）
- `bugfix-investigate-0001.md`（Bug 调查工作流 #1）
- `review-audit-0001.md`（审查工作流 #1）

## 固化工作流的来源

固化模板由以下三种途径沉淀而来：
1. **用户明确指示**："这个工作流以后就这么走" → 直接固化
2. **动态候选被采纳**：用户从 ≤3 个动态候选中选了某一条 → 自动固化
3. **重复任务识别**：同一类任务被动态编排 ≥3 次 → 主 agent 主动建议固化

## 强制加载清单

**主 agent 启动时必须加载**：
1. 本 README（理解双模式引擎）
2. `_intent-router.md`（意图判别规则）
3. `_template.md`（schema）
4. `INDEX.md`（已固化模板索引，用于阶段 1 粗筛）

**动态加载**：根据阶段 1 粗筛结果，按需加载命中的具体工作流模板。

**不得遗忘**——此清单已标注在 `~/.mavis/agents/mavis/memory/MEMORY.md` 中作为"项目必读项"。
